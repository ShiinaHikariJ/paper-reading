# 5. 对 E3 失败原因的排查

## 5.1 首先检查 Autoencoder

目标域重建结果：

```text
normalized_time_mse = 0.0053529
log_spectral_mse    = 0.0268282
```

结论：

> Autoencoder 能较好地重建 1500 rpm few-shot，Autoencoder 不是主要问题。

因此排除了：

```text
源域Autoencoder完全无法适应目标域
```

这一假设。

---

## 5.2 检查真实信号特征

真实 20-shot 的类别特征：

| 特征 | Normal | Outer | Inner |
|---|---:|---:|---:|
| RMS | 0.1886 | 0.3090 | 0.2745 |
| Kurtosis | 3.9616 | 3.6966 | 4.2046 |
| Crest factor | 4.8872 | 4.4375 | 4.9167 |
| 高频能量比 | 0.7719 | 0.7200 | 0.8387 |
| 频谱重心/Hz | 4122.8 | 3551.7 | 4081.6 |
| 包络峭度 | 5.3750 | 4.3260 | 5.8169 |

真实数据中存在明显类别差异：

- Outer 频谱重心较低；
- Inner 高频能量较高；
- 三类冲击性和包络特征不同。

---

## 5.3 检查 E3 生成信号

E3 生成信号：

| 特征 | Normal | Outer | Inner |
|---|---:|---:|---:|
| RMS | 0.1699 | 0.2382 | 0.2208 |
| Kurtosis | 2.9968 | 3.0254 | 3.0216 |
| Crest factor | 3.7809 | 3.8404 | 3.8548 |
| 高频能量比 | 0.7768 | 0.7819 | 0.7826 |
| 频谱重心/Hz | 4024.9 | 4020.6 | 4035.6 |
| 包络峭度 | 3.2631 | 3.3354 | 3.3152 |

核心问题：

```text
三类高频能量几乎一样
三类频谱重心几乎一样
三类包络峭度几乎一样
三类峭度全部接近3
```

这表明：

> E3 生成器发生了明显的类别坍缩和条件失效。

模型生成的是一个通用、平滑、接近高斯分布的振动信号，而不是三类具有不同冲击和频谱特征的故障信号。

# 6. E2/E3-v1 文本条件的问题

原始文本条件方式为：

```text
固定模板提示词
→ 随机初始化Token Embedding
→ 所有Token平均池化
→ FiLM注入扩散网络
```

这套方案存在多个问题。

## 6.1 文本编码器没有预训练语义

E2 只有 60 个目标样本，无法从头学会：

```text
outer race fault
inner race fault
high-frequency energy
impulsiveness
```

等术语的真正语义。

---

## 6.2 类别词被大量公共词稀释

三类描述的大多数词相同。

平均池化后：

```text
normal
outer race fault
inner race fault
```

等核心类别词容易被公共词淹没。

扩散模型因此可以忽略文本条件。

---

## 6.3 特征等级语义不统一

原始版本在不同类别、不同转速中分别计算：

```text
low
medium
high
```

因此相同的 `high` 在不同域中可能对应不同绝对范围。

源域和目标域的描述语义并不稳定。

---

## 6.4 E3 目标微调过强

原设置：

```text
源域扩散：20 epochs，lr=2e-4
目标微调：1200 epochs，lr=2e-4
```

目标域只有 60 个样本，却进行了大量高学习率更新。

这可能造成：

```text
源域预训练知识被目标域覆盖
```

---

## 6.5 生成样本比例过高

原始下游训练：

```text
每类真实20个
每类生成100个
```

生成数据占比约：

```text
83.3%
```

当生成质量差时，生成信号会直接淹没真实数据。

---

# 7. 今天构建的 Gen-v2 方案

为解决上述问题，构建了 Gen-v2。

条件设计改为：

```text
显式类别Embedding
+
显式转速Embedding
+
外部预训练文本Embedding
+
扩散时间步Embedding
```

同时增加：

```text
900/1500 rpm共享特征阈值
目标域低学习率、短微调
每类生成3倍候选再筛选
真实/生成1:1采样
```

设计了四组实验：

| 实验 | 源域预训练 | 文本Embedding |
|---|---:|---:|
| G0-direct | 否 | 否 |
| E2-v2 | 否 | 是 |
| G0-source | 是 | 否 |
| E3-v2 | 是 | 是 |

# 8. 今天为什么先跑了 G0

## 8.1 G0 的真实定位

G0 使用：

```text
类别ID + 转速ID
→ 条件扩散模型
```

G0 不包含：

```text
大模型
API
提示词
文本描述
语义Embedding
```

所以：

> G0 不是 Gen 方法本身。

G0 只是诊断基线，用来回答：

> 如果给扩散模型最明确的类别 ID，它能不能生成具有类别差异的信号？

---

## 8.2 G0 的价值

如果 G0 都完全失败：

```text
说明扩散骨干或条件注入有问题
```

如果 G0 有效：

```text
说明类别条件能起作用
后续可以继续加入文本语义
```

因此 G0 是排错步骤，不是最终研究贡献。

# 9. G0 运行过程中遇到的问题

生成阶段出现报错：

```text
RuntimeError:
Can't call numpy() on Tensor that requires grad.
Use tensor.detach().numpy() instead.
```

原因：

```python
autoencoder.decode(latent)
.cpu()
.numpy()
```

仍带有梯度记录。

修复为：

```python
autoencoder.decode(latent)
.detach()
.cpu()
.numpy()
```

该错误只发生在生成阶段，不影响已经完成的 Autoencoder 和扩散训练。

# 12. 今天最重要的认识修正

今天一度把 G0 当成了 Gen-v2 的主要方向。

后来重新明确：

> G0 与 Gen 的关系只是诊断对照，不是 Gen 方法本身。

真正的 Gen 应该包含：

```text
信号统计特征
→ 大模型生成或整理技术描述
→ 预训练文本模型生成语义Embedding
→ 文本条件控制扩散生成
```

G0 没有任何文本语义，因此不能回答：

```text
Gen是否有效
大模型提示词是否有效
API文本Embedding是否有效
```

它只能回答：

```text
类别条件扩散骨干能否工作
```

# 14. 当前阶段的正式结论

### 结论1：数据划分和下游评估流程正常

E0、E1 和测试集表现稳定。

主要表现：

```text
峭度接近3
```
